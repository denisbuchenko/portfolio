### Визуальные фичи и шейдеры (как это сделано)

Этот документ фиксирует **техники визуализации** в проекте: какие шейдеры используются, где именно они подключены и какие эффекты дают. Здесь **нет разбора кода построчно** — только принципы и пайплайны.

---

### Где живут шейдеры

Все GLSL-файлы лежат в одном месте: `src/shaders/`.

Далее они подключаются в TypeScript как raw-строки (через импорт с `?raw`) и передаются в `THREE.ShaderMaterial`.

---

### Итог: что сейчас есть в проекте

Сейчас в репозитории живут 3 “проекта”, доступные через launcher:

- **Частицы** — GPU‑сцена (Three.js + WebGL2), треки/paint слой.
- **Пазл** — 2D логика пазла + рисование трёхканальной маски и отображение по bit‑маске.
- **Фрукты** — отдельный debug‑проект для проверки рендера glTF‑моделей еды (для будущего использования в пазле как фон).

---

### Пазл: модель, маски, bit‑цвета и взаимодействия

#### 1) Модель пазла (4×4)

Пазл строится как сетка 4×4. Для каждого кусочка хранится:

- **Координаты в сетке** (`row/col`) и `id`
- **Соседи** (`neighbors.top/right/bottom/left`)
- **Тип соединения по каждой стороне**: `flat / tab / blank`
- **Ключ грани** (`key`) — идентификатор общей грани для пары соседей
- **Вариации формы** (`v0/v1`) — чтобы выемки/выпуклости не выглядели одинаковыми
- **`cellTopRightSrcPx`** — координаты верхнего‑правого угла клетки в исходной картинке (в px)

Файлы:
- `src/projects/puzzle/types.ts`
- `src/projects/puzzle/model.ts`

#### 2) Маска кусочка (bezier Path2D) и вырезание изображения

Контур кусочка строится процедурно как `Path2D` с bezier‑дугами по граням (flat/tab/blank).  
Дальше для каждого кусочка создаётся свой `ImageBitmap`: на offscreen‑canvas делается `ctx.clip(path)` и рисуется нужный участок исходной картинки.

Файлы:
- `src/projects/puzzle/path.ts`
- `src/projects/puzzle/pieceImage.ts`

#### 3) Стыковка кусочков (snap) и “группы”

После пристыковки кусочки объединяются в **группу**. При перетаскивании любого кусочка группы двигается вся группа.  
Снап проверяется только по “правильным” соседям из модели, а позиция вычисляется через известный `cellPx` (±1 клетка).

Файлы:
- `src/projects/puzzle/app/groups/groupSystem.ts`
- `src/projects/puzzle/app/groups/snapper.ts`

#### 4) Рисование (3 канала) → 8 цветов

Есть 3 “краски”: R/G/B. Каждая пишет только свой канал.  
Комбинации каналов дают **8 состояний**:

- `0` = (0,0,0) — “пусто”
- `1` = (1,0,0)
- `2` = (0,1,0)
- `4` = (0,0,1)
- `3` = (1,1,0)
- `5` = (1,0,1)
- `6` = (0,1,1)
- `7` = (1,1,1)

След реализован как 3 независимые полилинии (каждая со своим лимитом длины “змейкой”), рисуются в `paintCanvas` через `lighter`, чтобы каналы складывались.

Файл:
- `src/projects/puzzle/app/paint/paintSystem.ts`

#### 5) Маска влияет на видимость пазлов

Для каждого кусочка есть `maskBits` (0..7). Он виден только в тех местах экрана, где `bits(mask(x,y)) == maskBits`.

- Для CPU‑хиттестов маска downsample’ится (256×256), и `maskBitsAt(x,y)` очень быстро решает “можно ли тут хватать/тащить”.

Файл:
- `src/projects/puzzle/PuzzleProject.ts`

#### 6) Рендер пазла: Three.js + шейдеры‑маски

Пазл отрисовывается через Three.js (WebGL), потому что нужен быстрый пиксельный тест по маске:

- **Фоны** (7 шт.) — fullscreen quads, каждый рисуется только внутри своего `bits`:
  - `src/shaders/puzzleBgMasked.frag.glsl`
- **Кусочки пазла** — текстурные quads, каждый виден только внутри своего `maskBits`:
  - `src/shaders/puzzlePieceMask.frag.glsl`

Файл‑оркестратор:
- `src/projects/puzzle/app/render/puzzleRenderer.ts`

### Фрукты (debug‑проект): рендер glTF‑ассетов на экране

Это отдельный проект, чтобы независимо от пазла убедиться, что 3D‑ассеты:
- корректно загружаются,
- адекватно освещаются,
- не имеют артефактов по глубине,
- можно анимировать движение.

Что делает проект:

- Загружает glTF по URL из конфига (сейчас используется `CONFIG.puzzle.background3d.gltfUrl`)
- Выбирает “базовые” узлы по имени (`apple`, `banana`, …) без underscore
- Клонирует, центрирует и нормализует размер (через Box3)
- Назначает простые материалы `MeshLambertMaterial` (дёшево и “мультяшно”)
- Ставит ортокамеру с адекватным `near/far` и корректным фрустумом на весь экран
- Случайно расставляет и двигает объекты по экрану

Файлы:
- `src/projects/fruits/foodCatalog.ts`
- `src/projects/fruits/FruitsProject.ts`

### Карта шейдеров → модулей

- **`src/shaders/gasPoints.vert.glsl` / `src/shaders/gasPoints.frag.glsl`**
  - **Где используется**: `src/particles/gasPoints.ts`
  - **Что даёт**: отрисовка и анимация частиц (позиции считаются процедурно в vertex shader).

- **`src/shaders/fullscreenQuad.vert.glsl`**
  - **Где используется**: все fullscreen-проходы (`TrailComposer`, `PaintLayer`)
  - **Что даёт**: единый vertex для “экранного квадрата”, чтобы в фрагментных шейдерах думать только про `vUv`.

- **`src/shaders/trailsFade.frag.glsl`**
  - **Где используется**: `src/app/particleApp/TrailComposer.ts`
  - **Что даёт**: затухание/размытие буфера следов частиц (trail accumulation).

- **`src/shaders/texturePresent.frag.glsl`**
  - **Где используется**: `TrailComposer` (показ trail-RT), `PaintLayer` (похожая задача была раньше)
  - **Что даёт**: “показать текстуру как есть” (fullscreen blit).

- **`src/shaders/paintFade.frag.glsl`**
  - **Где используется**: `src/app/particleApp/PaintLayer.ts`
  - **Что даёт**: подготовка буфера рисования перед штампами — copy + мягкий blur (и опциональный decay, если включать).

- **`src/shaders/paintStamp.frag.glsl`**
  - **Где используется**: `PaintLayer`
  - **Что даёт**: штамп кисти в **маску плотности** (пишем в alpha), без “запекания” финального цвета.

- **`src/shaders/paintPresent.frag.glsl`**
  - **Где используется**: `PaintLayer`
  - **Что даёт**: превращает накопленную маску (alpha) в финальный “живой” вид: рваный контур, перетекание, переливы цвета, пульсирующее свечение.

- **`src/shaders/puzzleBgMasked.frag.glsl`**
  - **Где используется**: `src/projects/puzzle/app/render/puzzleRenderer.ts`
  - **Что даёт**: заливка фона только внутри нужного `bits` по маске.

- **`src/shaders/puzzlePieceMask.frag.glsl`**
  - **Где используется**: `src/projects/puzzle/app/render/puzzleRenderer.ts`
  - **Что даёт**: показ кусочков пазла только внутри их `maskBits` по маске.

### Частицы: движение и режимы (процедурная анимация)

Частицы не хранят состояние в текстурах позиций/скоростей. Вместо этого их координаты вычисляются **детерминированно** из:
- `uv` (уникальный “ID” частицы),
- `uTime`,
- набора uniform-параметров режимов.

Это даёт:
- стабильность (нет накопления ошибок),
- предсказуемость (одна и та же частица всегда имеет тот же seed),
- простое плавное смешивание режимов.

#### Режимы
- **Свободный газ (Mode = -1)**: “летающее облако” без хранения состояния (всё детерминировано).
  - **Seed per particle**: из `uv` делаются стабильные случайные числа (каждая частица уникальна, но воспроизводима).
  - **Стартовая позиция**: равномерно по области `uBounds` (world space).
  - **Скорость**: берётся в *пикселях/сек* (`speedPxMin/max`) и переводится в world units через `uPixelsPerWorld` → движение выглядит одинаково на разных экранах.
  - **Drift field**: небольшой процедурный “ветер/турбулентность” добавляет ощущение газа.
  - **Bounce without teleport**: позиция “отражается” в пределах `uBounds` без скачков (эффект как шарик в коробке, но детерминированный).

- **Аттрактор (Mode = 0)**: перевод части частиц на орбиту вокруг курсора/пальца.
  - **Плавное включение**: сила эффекта сглаживается во времени (в JS), чтобы не было резкого переключения.
  - **Захват по радиусу**: влияние зависит от дистанции до центра (мягкий falloff).
  - **Реконструкция стартового угла**: так как мы не храним state, “угол на старте” восстанавливается из положения частицы на момент включения режима (по `uAttractorStartTime`).

- **Сплайн‑путь (Mode = 1)**: частицы “едут” по траектории.
  - **Пер‑частичный сдвиг фазы**: параметр кривой \(t\) у каждой частицы свой (из `uv`), чтобы движение не было синхронным.
  - **Движение по параметру**: \(t = fract(phase + uTime * uBezierTimeScale + uBezierPhaseOffset)\). Это движение не по длине дуги, но очень дешёвое и стабильное.
  - **Источник траектории**:
    - **Кубическая Безье** (контрольные точки `uBezierP0..P3`),
    - или **произвольная SVG‑линия**, заранее сэмплированная по длине и загруженная как **1D‑текстура пути** (каждый texel хранит world‑координату точки).
  - **Сэмплинг 1D‑пути**: берём два соседних сэмпла и интерполируем — получается гладкая траектория при небольшом числе точек.
  - **Jitter**: у каждой частицы постоянный детерминированный оффсет (радиус `uBezierJitterRadius`) — “толщина” потока вокруг линии.
  - **Плавное включение**: позиция смешивается через `uBezierActive`, что даёт мягкий переход “газ ↔ сплайн”.

### Следы от частиц (trail accumulation)

Следы реализованы как отдельный слой с **ping-pong render targets**:

- **Fade pass** (`trailsFade.frag.glsl`): затухание предыдущего кадра + лёгкий blur, чтобы хвост был мягким.
- **Stamp pass**: поверх буфера штампуются текущие частицы (рендер сцены с точками).
- **Present** (`texturePresent.frag.glsl`): вывод буфера следов на экран.

Техника даёт “кометные хвосты” без хранения истории каждой частицы.

---

### Рисование: “плавающая клякса” (paint layer)

Это отдельный слой **под частицами**. Он тоже сделан через ping-pong RT, но с другой идеей:

#### 1) Накопление хранит только маску (плотность)
`paintStamp.frag.glsl` пишет **только alpha** (плотность “чернил/воды”). Это позволяет:
- сохранять рисунок персистентно (не исчезает),
- делать эффект “живым” не разрушая накопленные данные.

#### 2) “Жизнь” кляксы делается при отображении
`paintPresent.frag.glsl` использует несколько техник:
- **маска смещения (warp / flow field)**: анимированный Perlin‑подобный шум создаёт поле смещения, которым искажаются UV выборки базовой маски — клякса “перетекает” по краю;
- **шумный порог контура**: порог/ширина контура модулируются шумом, создавая “ломаность” и динамику края;
- **переливы цвета**: палитра ограничена голубым/синим/почти белым, а оттенок плавает от шума и времени;
- **пульсирующее свечение**: дополнительный glow вокруг активной зоны.

#### 3) Blur без выветривания
`paintFade.frag.glsl` используется как “copy + blur” перед штампом: убирает ступеньки и склейку сегментов, при этом рисунок не обязан исчезать (decay можно держать равным 1).

### Параметры “плавающей кляксы” (важно)

Все ключевые ручки находятся в `src/config.ts` (строки 29–46).

#### Текущие значения (именно они дают “эффект воды”)

Значения ниже — это **фактические** числа из `src/config.ts` (29–46) на текущий момент:

- **База кисти/пятна**
  - `paintHalfLife = 0.85`
  - `paintRadiusCssPx = 21`
  - `paintSpacingCssPx = 10`
  - `paintStampStrength = 0.4`
  - `paintNoiseScale = 50.0`
  - `paintEdgeAmp = 0.5`
  - `paintEdgeSoftness = 1.0`
  - `paintGlowIntensity = 1.6`
  - `paintPulseSpeed = 2.2`

- **“Плавающая клякса” (warp + контур)**
  - `paintWarpScale = 2.0`
  - `paintWarpSpeed = 0.85`
  - `paintWarpAmp = 0.1`
  - `paintContourThreshold = 0.08`
  - `paintContourWidth = 0.085`
  - `paintContourNoiseAmp = 1.0`

#### Базовые параметры кисти и “воды”
- **`paintRadiusCssPx`**: радиус кисти (ощущается как “палец”).
- **`paintSpacingCssPx`**: шаг штампов по траектории (меньше = более непрерывная линия).
- **`paintStampStrength`**: сила штампа (насколько быстро “набирается” плотность).
- **`paintNoiseScale`**: масштаб шума визуализации (детализация/частота “рваности” и переливов).
- **`paintEdgeAmp`**: сила влияния шума на край.
- **`paintEdgeSoftness`**: мягкость перехода внутри штампа (сама маска).
- **`paintGlowIntensity`**: интенсивность свечения.
- **`paintPulseSpeed`**: скорость пульса.

#### Параметры именно “плавающей кляксы” (warp + контур)
Это те настройки, которые создают ощущение **воды/текучести**:
- **`paintWarpScale`**: масштаб поля смещения (flow). Меньше = крупнее “волны”, больше = мельче детали.
- **`paintWarpSpeed`**: скорость перетекания поля.
- **`paintWarpAmp`**: амплитуда UV‑смещения (главный “двигатель” текучести по краю).
- **`paintContourThreshold`**: базовый порог контура (формирует толщину/плотность видимой зоны).
- **`paintContourWidth`**: ширина перехода контура (меньше = резче и “чернильнее”).
- **`paintContourNoiseAmp`**: насколько шум рвёт/двигает порог (делает край ломаным и живым).
